How it all connects — in short

  • Command chain: the dashboard and CLI talk to SystemController (logic and safety), which drives IOController (the drivers). Every write to a component is read back and verified (set & verify).
  • Two I2C buses: I2C0 carries the control side (PCA9685, TCA6424A, U6, U8) and I2C1 the sensing side (U7, U10). All bus access is serialized by a single lock.
  • LED ↔ fan: turning a UV LED on automatically starts its mapped fan per components.json — no code changes needed.
  • Heater is doubly interlocked: in software (its fan must spin above 100 RPM + thermistor must be valid, rechecked every 10 seconds) and in hardware (relay K3, energized only while the door is closed).
  • Door = three guards: the external sensor line physically disconnects K1/K2/K3; GPIO27 only listens to the line and reports to software; SystemController forces everything off the moment the door opens and blocks re-enabling.

Open items from the HSI (verify before relying on these)

  • GPIO17 (wavelength routing): no net with this name was found on any schematic sheet — the board may never have been updated for the move from GPIO27. A floating pin would return a false "verified" read-back. Two stale comments in the code still say GPIO27.
  • Door sensor polarity: working assumption: door closed = HIGH. Once confirmed, components.json open_level must change from 1 to 0 (the current value encodes the opposite polarity: safe but inoperable).
  • Door sensor output type: if it is a contact/open-collector with a weak pull-up, the relay LEDs will clamp the line (~1.2V) — the SSRs won't enable and the Pi will read LOW even with the door closed. Bench check required (measure at J17.2, pinctrl get 27, continuity through K1).
  • FAN1–6_ONOFF: the expander's fan-enable bits are not mapped to any physical fan and are not driven by the automatic control.
  • Pressure: the dashboard reads SENSOR1_A1 with a placeholder formula; the dedicated pressure sensor (ABP2, I2C 0x28) from the old HSI does not exist in the code — confirm what is actually populated on the board.
  • Light sensors LIGHT1/2: no visible front-end circuit in the schematic — the sheet may be missing or the design may have changed.
  • LED_SWITCH (P05) and LED_ANALOG5 (ch15): mapped but unused by any code — confirm their purpose before use.

On the board but not in the code yet

  • Full NFC: SPI (GPIO9/10/11) + CS (GPIO4) + INT (GPIO7) — only the reset pulse is implemented.
  • GPIO5: TCA6424A expander interrupt (the code polls instead of using the interrupt).
  • GPIO21: THER — thermal fault flag from U4 (moved from GPIO12 in the current revision, to be confirmed).
  • GPIO6 / GPIO18: BOFA_EN and BOFA_PRESENT detect — there is currently no way to enable the BOFA front end.
  • GPIO12: chamber RGB light (J3) — a hardware-PWM pin, not implemented.
  • UART (GPIO14/15 → J23): standard debug console.
